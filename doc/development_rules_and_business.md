# Magpie 协作规范与设计深化

## Git 分支与提交流程
1. **主干分支 (main)**：保持处于仅随时可发布的稳定状态。
2. **开发分支 (dev)**：所有合并到 main 的前置测试分支，功能开发全部在特性分支完成并 PR 到 dev 分支。
3. **特性分支 (feat/xxx)**：针对某一个具体的任务，例如 `feat/amadeus_api_ingestion`。
4. **提交信息规范 (Conventional Commits)**：
   - `feat:` 新增功能
   - `fix:` 修复 Bug
   - `chore:` 构建、基础设施变动
   - `docs:` 文档修改
   - `refactor:` 代码重构（无功能改变）

## 商业化潜能与“谁来买单”？（商业思维注入）
我们的核心不仅仅是一个冷冰冰的比价脚本，而是将**时间成本、情绪价值与信息差**进行变现。潜在买单方：
1. **异地恋高净值群体 / 极客情侣（C端）**：
   - *痛点*：每周查机票，消耗大量心力，还要对比周末天气和约会日历。
   - *买单点*：愿意为“只推送最完美的完美周末建议”支付 ￥9.9/月的订阅费。省下的机票钱远超订阅费，且省去了精神内耗。
2. **商旅频繁的小微企业 / 独立顾问（B端轻量级）**：
   - *痛点*：中小企业无专业行政比价。
   - *买单点*：系统自动为团队成员抓取“最低全服务航司价格”并在飞书中报告，企业老板或财务直接买单（包年）。
3. **周末特种兵 / 穷游大学生群体**：
   - *痛点*：极致追求价格绝对值低。
   - *买单点*：只要特定目的地（如：随时去成都吃火锅）低于心理防线，就立马买票。这种高频长线监控由于 OTA 杀熟而很难靠人力完成，可以作为一次性买断或按次分成的“抢票抽成”服务。

## 竞品/参考对象调研 (待用户深化)
在进入具体代码编写前，我需要你提供或由我们共同深挖以下开源系统及服务的实现细节：
- **现存竞品（商业 API 与系统）**：
  - 例如 Google Flights（目前唯一接近的推送引擎，但不支持极高频且不带 LLM 日程决策，更不支持微信推送）。
  - OTA 平台如携程自带的“降价提醒”（劣势：平台自身有意保持高价、推送有严重延迟且只涵盖单一系统，不支持如高铁 vs 机票的比价）。
- **可复用的开源（可行性评审）**：
  - `testerSunshine/12306`：对于价格聚合**不可用**（其架构专精于登录绕过和抢票验证码，极度笨重且价格信息维度低下，且有极高封号风险），建议舍弃，或者仅剥离其中的查票 JSON 接口参考。
  - `amadeus/amadeus-python`：**可用**。官方 SDK，但是我们需要在 Rust 端用 `reqwest` 自己写一层异步的高频包裹层以满足 Tokio 并发。
  - 各种机票反向代理如 `fli` 或 `fast-flights`：待评估，因为它们抓取的谷歌航班数据不一定支持极高并发。

本系统最宽广的护城河在于：**量化级别的价格捕捉速度** + **大模型多维决策（天气/日历情感连接）** + **私密化部署与零信任。**
