# 异地恋智能出行决策系统（Project Magpie）技术架构与演进路径深度调研报告

## 1. 总体系统架构与量化交易思维迁移
在现代个人计算与自动化技术的交汇点上，复杂的个人后勤规划正在经历向算法驱动与高度自治化的范式转移。异地恋通勤（如长跨度的南昌至北京航线或高铁网络）本质上是一个具有高度价格波动性、多维外部约束条件以及强时间敏感性的资源分配问题。

将金融领域的量化交易思维进行降维技术迁移，构建一个部署于个人服务器的全自动“量化买票”与度假决策智能体（Agent），代表了解决这一痛点的最优技术路径。该系统旨在通过突破传统在线旅行社（OTA）的平台信息壁垒，建立高频监控、时序预测、多模态上下文分析以及自动化触达的闭环体系。

系统架构被严格设计为四个高度解耦的模块：
1. **感知层**：负责高频数据摄取。
2. **记忆层**：负责状态存储与统计计算。
3. **决策层**：主导逻辑推理与外部工具调用。
4. **触达层**：负责工作流调度与终端交互。

这种微服务级别的解耦不仅有利于独立模块的快速技术迭代，也为未来通过自然语言生成代码（Vibe Coding）进行局部重构提供了极大的架构弹性。

## 2. 感知层：多模态交通数据高频摄取架构
数据摄取层是整个自治系统的基础设施，也是技术挑战最为集中的防御对抗前线。航空与铁路票价系统具有完全不同的定价逻辑与反爬虫防御机制。

### 2.1 航空票价数据摄取：官方网关与逆向工程协议
航空公司的动态定价网络与收益管理系统极其复杂。
- **官方 API**：Amadeus Self-Service API 提供了高度结构化的 JSON 数据流，其测试环境允许较高并发请求，适合高频价格快照采集。但可能存在部分廉航或特定航线的数据盲区。
- **逆向工程 API**：基于协议层逆向的开源工具（如 `fli`、`fast-flights`）性能优势显著，不仅零前端渲染延迟，还能直接解析底层 Protobuf/JSON 载荷，原生支持代理轮换接入。

### 2.2 航空数据反爬对抗：隐匿浏览与指纹伪装技术
面对现代网络应用防火墙（WAF，如 Cloudflare），传统的无头浏览器方案（如 Puppeteer Stealth）已逐渐失效。
- **新一代工具链**：推荐使用 Python 生态下的 `Nodriver`、`SeleniumBase UC`，或基于 Firefox 内核定制的 `Camoufox`。
- **代理与流量混淆**：结合住宅 IP（Residential IPs）代理池和高级绕过网关（如 ZenRows），以规避由于地理位置或 JA3 指纹暴露导致的风控。

### 2.3 铁路系统数据摄取：从图像识别到模型上下文协议 (MCP)
12306 铁路系统主要依赖高频 IP 封锁与复杂的图像验证码。
- **MCP 架构**：通过开源项目如 `12306-mcp`，为 LLM 提供标准化的 API 网关。该架构不仅封装了查票逻辑、签名生成，还支持跨站智能中转方案计算。
- **环境桥接**：结合 `mcp-chrome` 等工具，直接在用户日常浏览器进程中注入服务，继承真实用户登录状态，实现无感抓取。

### 2.4 数据源摄取策略对比矩阵

| 数据源类型 | 摄取技术路径 | 核心优势 | 潜在缺陷与对抗策略 |
|---|---|---|---|
| **国际与全网航空** | Amadeus API | JSON 结构化输出，合法稳定，测试环境配额高 | 缺乏部分廉航数据；需通过聚合平台补充 |
| **聚合比价引擎** | 逆向工程 (fli, fast-flights) | 零网页渲染延迟，直接解析底层载荷，速度极快 | 易受上游协议变更影响；需代理重试机制 |
| **特定 OTA 平台** | 新一代反检测浏览器 (Nodriver 等) | 完全模拟人类视觉渲染，可获取平台专属促销折扣 | 资源消耗巨大；需对齐 JA3 指纹与 IP 地理位置 |
| **国内铁路 (12306)** | 模型上下文协议 (12306-mcp) | 接口标准化，原生适配 LLM 智能体，支持中转计算 | 需部署于高信誉住宅网络；可结合浏览器扩展注入 |

## 3. 记忆层与时序分析层：超轻量级架构下的绝对低价探测
感知层获取的数据流需要长期的历史沉淀与数学统计，从而为量化预测提供数据基座。

### 3.1 嵌入式分析型数据库选型：列式存储的降维优势
- **SQLite 局限性**：在面对高频执行窗口操作、全表扫描的时间序列 OLAP 场景下，SQLite 作为行式数据库性能衰减严重。
- **DuckDB 优势**：采用列式存储与向量化查询执行引擎，极大地提升了滑动平均等统计运算的速度。同时与 Python Pandas/Arrow 生态无缝集成，降低了序列化损耗。

### 3.2 从均线回归到状态空间模型：Mamba 架构引入
*MVP 阶段*的移动平均（MA）策略易产生误报。系统第二阶段将引入基于状态空间模型（SSMs）的 **Mamba** 架构：
$$h_t = \bar{A} h_{t-1} + \bar{B} x_t$$
$$y_t = \bar{C} h_t$$
Mamba 能够以极低的显存占用，线性时间处理长达数月的购票历史序列，捕捉复杂的长程时间依赖性与节假日脉冲特征。

### 3.3 双网络概率时序预测：Mamba-ProbTSF 与置信区间
直接点预测无法量化机票价格的高度随机性。
- **Mamba-ProbTSF**：引入双网络结构，主网络预测价格走向，辅助网络对预测方差进行建模，输出动态的不确定性区间。
- **决策跃升**：只有当实际票价跌穿预测模型下轨的置信区间边界（Two-Sigma）时，系统才确认捕捉到“绝对低价”异常事件并触发提醒。

## 4. 决策层：基于大型语言模型的本地化多维推理大脑
仅仅拥有价格优势不足以构成完美的出行决策，需要 LLM 将其与外部上下文（天气、日程）进行高维融合。

### 4.1 本地化模型部署与多智能体路由架构
考虑到 API 成本与隐私问题，通过 Ollama 等容器在个人服务器上部署轻量化模型（如 Llama-3-Groq-Tool-Use）。
采用**多智能体路由架构（Multi-Agent Router）**：专门的“天气智能体”、“日程智能体”与“量化分析智能体”被路由节点统筹，合并生成最终决策，降低幻觉风险。

### 4.2 工具调用与结构化输出：JSON Schema 的强约束机制
使用 Function Calling 让 LLM 获取实时外部数据。强制模型在诸如天气查询时，遵循具有严格类型与字段约束的 JSON Schema 载荷。同时，利用最新框架对结构化输出的原生支持，让 LLM 的最终决策也能以可靠的 JSON 形式输出，便于后续的自动化流转。

### 4.3 提示词工程演进：思维链与休假决策推理栈
系统提示词设计需遵循严格的结构：
1. **角色定义**：精通成本核算与旅行体验优化的私人助理。
2. **上下文注入**：动态注入历史价格基准、Mamba 预测区间与目标地周末天气。
3. **推理任务约束 (CoT)**：强制逐步推理（如，先评估价格偏离度，再看天气是否存在一票否决权，最后得出性价比）。
4. **输出格式执行**：输出包含决策枚举项（BUY/HOLD）、置信度与 50 字以内自然语言摘要。

## 5. 触达与自动化层：全天候无感运行的编排拓扑
系统需要稳定、可观测的神经中枢进行统一调度，并无界触达用户的移动终端。

### 5.1 容器化工作流编排：基于 n8n 的微服务调度
- 使用 n8n 代替 Python 的定时脚本，提供可视化的节点编排流水线。
- 通过 Docker Compose 部署，结合 PostgreSQL 及反向代理（Traefik），实现持久化与内网穿透能力。
- 设计含有条件节点的 Workflow：仅当解析到 `{"action_decision": "BUY"}` 时才触发下发通知。

### 5.2 异步消息分发网络：绕过公网 IP 的微信触达机制
- **ServerChan³ / WxPusher**：成熟的托管式推送网关，提供即开即用的微信消息投递能力。
- **nanobot (超轻量级本地推送)**：基于 WebSocket 长连接机制，无需公网 IP 及复杂端口监听，极其适合高隐私要求的自我部署场景。

## 6. 演进路线：从被动监控到全自动闭眼出游

### 阶段 1 (v0.1) - MVP 核心逻辑闭环构建
*   利用逆向脚本按固定频率抓取票价，存入数据库。
*   基于 Pandas 进行移动均线（MA Crossover）计算。
*   n8n 定时触发，若出现断崖下跌，通过 WxPusher Webhook 发送基础提醒。

### 阶段 2 (v1.0) - 多模态智能上下文融合
*   预测模型升级至本地 Mamba-ProbTSF 分析出价格极值下行边界。
*   接入本地 LLM，进行多智能体路由与 API Function Calling（查询天气、日历）。
*   推送富文本结合的微度假报告至微信。

### 阶段 3 (v2.0) - 完全自治代理与执行器注入
*   在此阶段系统试图获取用户实际的购买权限（如通过 mcp-chrome 辅助下单）。
*   从决策参考进化到“发现即锁定”的完全自治体，实现极致自动化，甚至替用户垫付款项并推送最终账单（需考虑极高的安全对抗及合规问题）。
